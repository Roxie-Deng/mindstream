## 目标锚定

每次讨论前先对齐目标：
1. 可部署
2. 可写简历
3. 可回答面试

**Why:** 用户是转行求职者，时间有限，必须聚焦核心价值

**How to apply:** 任何讨论如果偏离这三个目标，主动拉回来

---

## 项目推进流程

按以下顺序逐步推进，每次只做一个模块：

```
需求分析 → 模块设计 → 数据库设计 → API设计 → 代码实现
```

---

## 每个模块完成后输出

1. **设计思路** — 为什么这么做，不这么做有什么问题
2. **技术选型** — 选了什么，为什么
3. **面试问题** — 这个模块相关的面试题和答案
4. **项目文档** — 更新到 SPEC.md
5. **面试笔记** — 追加到 LOG.md

---

## 决策原则

- **过度设计 → 提醒**：如果开始画大饼、预支未来需求，主动喊停
- **面试价值不高 → 指出**：如果某个功能对简历和面试没帮助，说清楚
- **优先工程合理性**：先跑通，再优化
- **不生成整个项目**：一次只推进一个模块
- **多租户现在不做**：只在所有表加 `user_id INT DEFAULT 1`，日后好改

---

## 注意事项

- 不要直接生成整个项目
- 每次只实现一个模块
- 确认一个模块的设计后再进入下一个
